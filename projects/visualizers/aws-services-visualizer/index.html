VPC & Networking

A Virtual Private Cloud (VPC) is your isolated network in AWS. It contains subnets (public & private), route tables, Internet Gateways (IGW), NAT Gateways, and security layers. Every AWS resource you deploy lives inside a VPC.

🌐 VPC Architecture Diagram

🔄 NAT Gateway
VPC 10.0.0.0/16
💡 Key Concept: Public subnets have a route to the IGW (resources can reach the internet directly). Private subnets route through a NAT Gateway — resources can initiate outbound connections but can't be reached from the internet.

🛡️ Security Groups vs NACLs

🔒 Security Groups

  • Stateful — return traffic auto-allowed
  • Instance-level (attached to ENI)
  • Allow rules only (no deny rules)
  • All rules evaluated before decision
  • Default: deny all inbound, allow all outbound

🚧 Network ACLs (NACLs)

  • Stateless — must allow return traffic explicitly
  • Subnet-level firewall
  • Allow AND deny rules
  • Rules processed in order (lowest number first)
  • Default: allow all in/out
📌 Interview Tip: Security Groups = stateful instance firewall. NACLs = stateless subnet firewall. Use SGs for fine-grained control per resource, NACLs for broad subnet-level blocking (e.g., block a known malicious IP range).

📋 VPC Key Terms

CIDR Block
IP range for your VPC (e.g., 10.0.0.0/16 = 65,536 IPs). Cannot be changed after creation.
Route Table
Rules that determine where network traffic is directed. Each subnet has one.
Elastic IP
Static public IPv4 address. Free when attached to a running instance; charged when idle.
VPC Peering
Private connection between two VPCs. Non-transitive — A↔B and B↔C does NOT mean A↔C.
VPC Endpoints
Private access to AWS services (S3, DynamoDB) without traversing the internet.
Transit Gateway
Central hub to connect multiple VPCs across accounts/regions. Solves peering complexity.

IAM — Identity & Access Management

IAM controls who (authentication) can do what (authorization) on which resources. It uses Users, Groups, Roles, and Policies — the foundation of AWS security.

👥 IAM Entities

🔐 Policy Simulator

Toggle policy rules to see how AWS evaluates access. Remember: an explicit Deny always wins over any Allow.

⚠️ Evaluation Logic: 1) Start with implicit Deny → 2) Check all policies → 3) If any explicit Deny → DENIED → 4) If any Allow → ALLOWED → 5) Otherwise → implicit DENY. Explicit Deny always wins.

📋 IAM Key Terms

IAM Role
Temporary credentials for services/users. No password — uses STS AssumeRole. Preferred over long-term keys.
STS
Security Token Service — issues temporary credentials. Used for cross-account access, federation.
Principle of Least Privilege
Grant only the minimum permissions needed. Start with zero access, add as required.
Trust Policy
Defines WHO can assume a role. Attached to the role itself (not the user).
Permission Boundary
Maximum permissions an IAM entity CAN have. Intersection of boundary and identity policy.

SQS — Message Queuing

Amazon SQS decouples components by placing messages in a queue. Producers send messages; consumers poll and process them. SQS supports Standard (best-effort ordering, at-least-once) and FIFO (strict ordering, exactly-once) queues.

📬 Queue Simulation

📤 Producer
📬 Queue
✅ Processed
💀 DLQ

Send messages, then process them. Messages that fail 3 times are moved to the Dead Letter Queue (DLQ).

⚖️ Standard vs FIFO

📨 Standard Queue

  • Nearly unlimited throughput
  • Best-effort ordering
  • At-least-once delivery (may duplicate)
  • Use: log processing, background jobs

📋 FIFO Queue

  • 300 msg/sec (3000 with batching)
  • Strict ordering within message group
  • Exactly-once processing
  • Use: financial transactions, order processing

📋 SQS Key Terms

Visibility Timeout
After a consumer reads a message, it's hidden from other consumers for this duration (default 30s). If not deleted, it reappears.
Dead Letter Queue
Messages that fail processing N times are moved here. Set via maxReceiveCount in redrive policy.
Long Polling
Wait up to 20s for messages. Reduces empty responses and API costs. Set WaitTimeSeconds > 0.
Message Group ID
FIFO only — messages with same group ID are delivered in order. Different groups can be processed in parallel.
Deduplication ID
FIFO only — prevents duplicate messages within a 5-minute window.
Max Message Size
256 KB. For larger payloads, store in S3 and send a pointer (SQS Extended Client Library).

SNS — Pub/Sub Notifications

Amazon SNS is a pub/sub messaging service. A topic acts as a channel — publishers send messages to topics, and all subscribers (Lambda, SQS, HTTP, email, SMS) receive a copy. It's the foundation for fan-out architectures.

📡 Fan-Out Demo

📬 SQS → Inventory
⚡ Lambda → Analytics
🔗 HTTP → Partner API
Fan-Out Pattern: One publish → N deliveries. Each subscriber processes independently. If one fails, others are unaffected. SNS retries failed deliveries.

📋 SNS Key Terms

Topic
A named channel for messages. Supports up to 12.5M subscriptions per topic.
Message Filtering
Subscribers can set filter policies to receive only matching messages. Reduces processing waste.
FIFO Topic
Strict ordering + deduplication. Only SQS FIFO queues can subscribe.
Message Attributes
Key-value metadata sent with each message. Used by filter policies for routing.
Delivery Protocols
SQS, Lambda, HTTP/S, Email, SMS, Kinesis Data Firehose, mobile push (APNs, FCM).

SNS + SQS — Fan-Out Pattern

The SNS → SQS fan-out is the most common event-driven pattern in AWS. SNS distributes the message; each SQS queue buffers it for independent processing. This gives you the reliability of SQS with the broadcast of SNS.

Why SNS + SQS? SNS alone doesn't retry or buffer. If a subscriber is down, the message is lost. Adding SQS between SNS and the consumer provides durability (messages persist up to 14 days), retry logic, and DLQ support.

🏗️ When to Use Which

📬 SQS Only

  • One producer → one consumer
  • Task queue / work distribution
  • Rate smoothing / buffering
  • Order processing pipeline

📢 SNS + SQS

  • One event → multiple downstream systems
  • Microservice event broadcasting
  • Each consumer processes independently
  • Decoupled, fault-tolerant architecture

ECS & EKS — Container Orchestration

ECS (Elastic Container Service) is AWS's native container orchestrator. EKS (Elastic Kubernetes Service) is managed Kubernetes. Both run Docker containers but differ in complexity, portability, and ecosystem.

⚖️ ECS vs EKS

🐳 ECS

  • AWS-native, simpler to set up
  • Task Definitions define containers
  • Fargate = serverless (no EC2 to manage)
  • Deep AWS integration (ALB, CloudWatch)
  • Best for: AWS-centric teams, simpler workloads

☸️ EKS

  • Managed Kubernetes (CNCF standard)
  • Pods, Deployments, Services
  • Portable across clouds / on-prem
  • Rich Kubernetes ecosystem (Helm, Istio)
  • Best for: multi-cloud, complex microservices

📋 Container Key Terms

Task Definition
ECS blueprint for containers: image, CPU, memory, ports, env vars. Like a docker-compose file.
Service
Maintains desired count of tasks. Auto-replaces failed tasks. Integrates with ALB for load balancing.
Fargate
Serverless compute for containers. No EC2 management. Pay per vCPU+memory per second.
ECR
Elastic Container Registry — private Docker image repo. Integrates with ECS/EKS for image pulls.
Pod (EKS)
Smallest deployable unit in Kubernetes. Contains one or more containers sharing network/storage.
Service Mesh
App Mesh (ECS) or Istio (EKS) — manages service-to-service communication, observability, traffic routing.

🧊 Cold Start Explained

Cold Start: First invocation after idle → AWS must: 1) Provision a container, 2) Download code, 3) Initialize runtime, 4) Run your handler. This adds 100ms–10s latency. Subsequent calls reuse the warm container (warm start = ~1ms overhead).
Mitigation: Use Provisioned Concurrency (keeps N instances warm), minimize package size, use lightweight runtimes (Python/Node over Java), put initialization code outside the handler function.

📋 Serverless Key Terms

Invocation Types
Synchronous (API GW), Asynchronous (S3/SNS events), Poll-based (SQS/DynamoDB streams).
Concurrency
Default 1000 concurrent executions per account per region. Reserved concurrency caps per function.
Layers
Reusable packages (libs, SDKs) shared across functions. Max 5 layers, 250MB total unzipped.
Destinations
Route async invocation results (success/failure) to SQS, SNS, Lambda, or EventBridge.
API GW: REST vs HTTP
REST API: full-featured (caching, WAF, usage plans). HTTP API: simpler, cheaper, faster. Use HTTP API unless you need REST extras.
Lambda@Edge
Run Lambda at CloudFront edge locations. Use for auth, URL rewrites, A/B testing at the CDN layer.

DynamoDB — NoSQL Database

Amazon DynamoDB is a fully managed, serverless NoSQL database with single-digit millisecond performance at any scale. It uses partition keys and optional sort keys for data distribution and supports Global Secondary Indexes (GSI) and Local Secondary Indexes (LSI).

🏗️ Table Designer

Select a schema pattern to see how DynamoDB structures the data.

💡 Single-Table Design: Store multiple entity types (Users, Orders, Products) in one table using carefully crafted PK/SK patterns. Reduces costs and enables complex queries with a single request.

⚡ Capacity Modes

🔄 On-Demand Mode

  • Pay-per-request — no capacity planning
  • Auto-scales instantly to any traffic level
  • Best for: unpredictable workloads, new tables
  • Cost: ~$1.25/million write, ~$0.25/million read
  • No throttling risk (within account limits)

📊 Provisioned Mode

  • Set RCU/WCU — predictable cost
  • Auto Scaling available (target utilization)
  • Best for: steady, predictable workloads
  • Cost: ~$0.00065/WCU/hr, ~$0.00013/RCU/hr
  • Reserved Capacity = up to 77% savings

🔍 GSI vs LSI — Index Visualizer

See how indexes unlock different query patterns on the same data.

📌 Key Difference: GSI = new partition key + optional sort key. Can be added anytime. Has own RCU/WCU. LSI = same partition key, different sort key. Must be created with the table. Shares RCU/WCU with base table. Max 5 LSIs per table.

📋 DynamoDB Key Terms

Partition Key
Hash key that determines which partition stores the item. Must be unique (alone) or unique with sort key.
Sort Key
Range key enabling range queries within a partition. Enables begins_with, between, >, < queries.
RCU / WCU
Read/Write Capacity Units. 1 RCU = 1 strongly consistent read/sec (4KB). 1 WCU = 1 write/sec (1KB).
DAX
DynamoDB Accelerator — in-memory cache. Microsecond reads. Drop-in compatible (same API).
TTL
Time-to-Live — auto-delete expired items at no cost. Set a timestamp attribute, DynamoDB handles deletion.
Transactions
ACID transactions across up to 100 items/25 tables. TransactWriteItems / TransactGetItems. 2x RCU/WCU cost.
PartiQL
SQL-compatible query language for DynamoDB. SELECT, INSERT, UPDATE, DELETE syntax.
Global Tables
Multi-region, multi-active replication. Sub-second replication. Requires Streams enabled.

Step Functions — Advanced Patterns

Beyond basic workflows, Step Functions enable Saga patterns for distributed transactions, parallel execution, Map state for batch processing, and callback patterns for human-in-the-loop workflows.

🔄 Saga Pattern — Distributed Transactions

Run the saga to see a multi-step transaction. "Fail Payment" triggers compensating actions (rollback).

📦 Reserve Inventory
💳 Process Payment
🚚 Schedule Shipping
📧 Send Confirmation
↩️ Compensating Actions
📦 Release Inventory
💸 Refund Payment
Saga Pattern: Unlike 2-phase commit, each step has a compensating action. On failure, Step Functions runs compensations in reverse order to undo completed steps. This ensures eventual consistency in distributed systems.

⚖️ Standard vs Express Workflows

📋 Standard Workflow

  • Duration: up to 1 year
  • Exactly-once execution
  • Full execution history in console
  • Pricing: per state transition ($0.025/1000)
  • Use: order processing, ETL, long-running

⚡ Express Workflow

  • Duration: up to 5 minutes
  • At-least-once (async) / at-most-once (sync)
  • Logs sent to CloudWatch
  • Pricing: per execution + duration ($1/million)
  • Use: IoT ingestion, streaming, high-volume

📋 Advanced Step Functions Key Terms

Saga Pattern
Distributed transaction with compensating actions. Each step's "undo" is triggered on failure.
Map State
Iterate over an array, running a sub-workflow per element. Inline (40 concurrency) or Distributed (10K+).
Callback Pattern
Task sends token to external system (e.g., human approval). Workflow pauses until SendTaskSuccess/Failure.
Activity Tasks
Worker-based polling model. External workers poll for tasks, process, and report results.
Idempotency
Critical for retries. Use idempotency keys to prevent duplicate side effects on re-execution.
Error Handling
Retry: exponential backoff with IntervalSeconds, MaxAttempts, BackoffRate. Catch: route to fallback state.

⚖️ REST API vs HTTP API

🌐 REST API

  • Full feature set
  • API Keys & Usage Plans
  • Response Caching
  • WAF Integration
  • Mapping Templates (VTL)
  • Edge-optimized endpoints

⚡ HTTP API

  • 71% cheaper, up to 60% faster
  • OIDC/OAuth 2.0 native auth
  • CORS support built-in
  • No usage plans / caching
  • No WAF support
  • Best for: simple serverless proxies

📋 API Gateway Key Terms

Lambda Authorizer
Custom Lambda function that controls access using bearer tokens or headers. Returns IAM policy.
Cognito Authorizer
Built-in auth using Amazon Cognito User Pools. Validates JWT automatically without writing code.
Usage Plan
Groups API keys. Sets rate limits (requests/sec) and quotas (requests/month). Monetize APIs.
Throttling (429)
Default 10,000 rps per region. Prevent overload. Burst limits handle traffic spikes.
Mapping Template
Uses VTL (Velocity Template Language) to transform JSON payload before/after integration.
Stage Variables
Env vars for API deployment stages (dev/prod). Often used to route to specific Lambda aliases.
WebSocket API
Stateful, bidirectional communication. Requires routes ($connect, $disconnect, $default, custom).

Combined Architecture Patterns

Real-world systems combine multiple AWS services. Here are 3 common patterns showing how services work together for web apps, event processing, and microservices.

🏗️ Architecture Decision Guide

Synchronous
Client waits for response → API GW + Lambda + DynamoDB. Simple, low latency.
Asynchronous
Fire-and-forget → SNS/SQS + Lambda. Decoupled, fault-tolerant, scalable.
Orchestration
Complex multi-step workflows → Step Functions. Visual, auditable, built-in error handling.
Choreography
Services react to events → EventBridge. Loosely coupled, independently deployable.
Batch Processing
Large dataset processing → S3 + Lambda/ECS + Step Functions. Event-driven batch.
