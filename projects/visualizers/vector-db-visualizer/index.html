Embeddings: Words to Vectors

Vector databases don't store text directly; they store embeddings — lists of numbers (vectors) that represent the meaning of the text. Words with similar meanings map to points close to each other in this mathematical space.

🧭 2D Embedding Space

Type words to see where they land in a simplified 2D space. Notice how similar words cluster together!

💡 High Dimensions: In reality, models like OpenAI's text-embedding-ada-002 create vectors with 1,536 dimensions, not 2! High dimensions capture rich nuance (grammar, tone, context).
02

Distance Metrics

To find "similar" things, we calculate the mathematical distance between their vectors. Common metrics include Cosine Similarity, Euclidean Distance (L2), and Dot Product.

Calculation

Try dragging the tips of Vector A (Amber) and Vector B (Cyan) to see how the metrics change.

🔑 Cosine vs Euclidean: Cosine similarity measures the angle between vectors (ignoring length). Euclidean measures straight-line distance. If you normalize vectors to length 1, Cosine and Euclidean rank results identically!
03

KNN vs ANN Search

K-Nearest Neighbors (KNN) checks every vector to find the closest matches. It's accurate but slow. Approximate Nearest Neighbors (ANN) uses indexes to skip points, trading a little accuracy for massive speedups.

🚀 Why ANN? At 1 million vectors, exhaustive KNN might take 500ms. ANN (like HNSW) can find the neighbors in <1ms. Vector DBs are fundamentally built around ANN indexing algorithms.
04

HNSW Algorithm (Under the Hood)

Hierarchical Navigable Small World (HNSW) is the most popular ANN index. It builds multiple layers of graphs. It starts searching in sparse top layers to make huge leaps, then drops to lower, denser layers to fine-tune the match.

🕸️ HNSW Layer Traversal

🔗 Small World Graph: Inspired by the "six degrees of separation". Short-range links connect close neighbors, while long-range links connect distant clusters, allowing extremely fast navigation across the graph.
05

Metadata Filtering

Often you want to combine semantic search with exact filters (e.g., "Find comfy shoes, but only size 10"). Vector DBs handle this via Pre-filtering or Post-filtering.

Method
How it works
Pros / Cons
Post-filtering
1. ANN search returns top K results.
2. Filter out items that don't match metadata.
✅ Fast search
❌ Missing results (if filter is strict, top K might all get filtered out)
Pre-filtering
1. Filter DB by metadata first.
2. Perform ANN search on the remaining subset.
✅ Accurate K results
❌ Breaks HNSW graph structure if too many nodes are removed
Single-Stage Filter
Traverse the ANN graph, but only evaluate nodes that match the metadata filter.
✅ Modern standard (Pinecone, Qdrant, Milvus)
✅ Highly accurate & efficient

Use Case: Document Chunking for RAG

Before storing documents in a vector DB, you must chunk them — split them into smaller pieces. Chunk size critically affects retrieval quality. Too large = noisy context. Too small = lost meaning.

✂️ Interactive Chunking Strategies

🎯 Best Practice: Use semantic chunking (split by topic boundaries) with ~200-500 token chunks and 10-20% overlap. Libraries like LangChain and LlamaIndex provide built-in recursive text splitters.
09

Use Case: Recommendation Engine

Vector DBs power modern recommendation systems. By embedding user preferences and items into the same vector space, finding "similar items" becomes a nearest-neighbor search.

🎬 Movie Recommendation Engine

Select a movie you like — the vector DB finds the most similar movies by embedding proximity.

🔗 How Netflix does it: Netflix embeds each user's viewing history and each movie's features into the same vector space. Finding recommendations = finding the nearest movie vectors to the user's preference vector. They process billions of vectors daily.

Use Case: Chatbot Memory

LLMs have a limited context window. By storing past conversations in a vector DB, chatbots can recall relevant history from thousands of messages — giving the illusion of long-term memory.
