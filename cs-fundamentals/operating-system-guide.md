## 3ï¸âƒ£ CPU Scheduling Algorithms

### FCFS (First Come First Serve)

```
Process   Arrival   Burst Time
  P1         0         24
  P2         1          3
  P3         2          3

Timeline:
|-------- P1 (24) --------|-- P2 (3) --|-- P3 (3) --|
0                        24           27           30

Problem: Convoy Effect (Short jobs wait for long jobs)
Average Waiting Time: (0 + 23 + 25) / 3 = 16
```

### Round Robin (Time Quantum = 4)

```
Process   Arrival   Burst Time
  P1         0         10
  P2         0          4
  P3         0          5

Timeline:
|P1|P2|P3|P1|P3|P1|
 4  4  4  4  1  2  â† Remaining burst handled in rounds

Fair for all processes, used in most modern OS
Time Quantum too high â†’ Becomes FCFS
Time Quantum too low â†’ Too many context switches
```

### Priority Scheduling

```
Process   Burst   Priority (lower = higher)
  P1        10          3
  P2         1          1   â† Runs first
  P3         2          2

Execution Order: P2 â†’ P3 â†’ P1

Problem: Starvation (low priority never runs)
Solution: Aging (increase priority over time)
```

### Race Condition Example

```java
// Two threads incrementing same counter:
// Expected: 2000, Actual: Random (1500, 1800, etc.)

int counter = 0;  // Shared variable

// Thread 1                    // Thread 2
for (i = 0; i < 1000; i++)    for (i = 0; i < 1000; i++)
    counter++;                     counter++;

// WHY? counter++ is NOT atomic:
//   1. Read counter
//   2. Increment
//   3. Write back
// Threads can interleave between these steps!
```

â€¢ Lock Timeout: Give up if can't acquire in X seconds
â€¢ Deadlock Detection: Detect cycles, kill one process

### Q3: Explain virtual memory and page fault.

```
ANSWER:
Virtual Memory:
- Each process gets illusion of large contiguous memory
- OS manages mapping of virtual â†’ physical pages
- Not all pages are in RAM (some on disk/swap)

Page Fault:
1. Process accesses virtual address
2. Page table says page is NOT in RAM
3. CPU generates interrupt (page fault)
4. OS loads page from disk to RAM
5. Update page table
6. Restart the instruction

TYPES:
â€¢ Minor: Page is in RAM (just not mapped)
â€¢ Major: Page must be loaded from disk (slow!)
```

### Q4: What is the difference between mutex and semaphore?

```
ANSWER:
MUTEX (Binary Lock):
- Only 0 or 1
- Owner thread must unlock it
- Used for: Protecting critical section
- Example: Only one thread writes to file

SEMAPHORE (Counting Lock):
- Can be any non-negative integer
- Any thread can signal (increment)
- Used for: Resource pools, producer-consumer
- Example: Database connection pool (max 10)
```

### Q5: Explain context switching overhead.

```
ANSWER:
Context Switch Steps:
1. Save CPU registers of current process
2. Save program counter and stack pointer
3. Update PCB (Process Control Block)
4. Load PCB of new process
5. Restore registers, PC, SP of new process
6. Flush TLB (Translation Lookaside Buffer)
7. Possible cache misses (cold cache)

OVERHEAD:
- Direct: Register save/restore (~1-10 Î¼s)
- Indirect: Cache/TLB misses (major overhead!)

WHY IT MATTERS:
- Too many context switches = wasted CPU time
- Virtual threads minimize this (user-space switching)
```

### Q6: What is thrashing?

```
ANSWER:
Thrashing = CPU spends more time swapping pages than executing

CAUSE:
1. Too many processes in memory
2. Working set > Physical RAM
3. Continuous page faults

SYMPTOM:
- High disk I/O
- Low CPU utilization
- System feels frozen

SOLUTION:
- Add more RAM
- Reduce number of processes
- Use working set model
```

### Q7: Explain the difference between preemptive and non-preemptive scheduling.

```
ANSWER:
NON-PREEMPTIVE (Cooperative):
- Process runs until it voluntarily yields
- Example: FCFS, Shortest Job First
- Risk: One process can hog CPU forever

PREEMPTIVE:
- OS can interrupt running process
- Based on timer interrupt or priority
- Example: Round Robin, Priority Scheduling
- Prevents starvation, fairer

Modern OS: Always preemptive (Windows, Linux, macOS)
```
